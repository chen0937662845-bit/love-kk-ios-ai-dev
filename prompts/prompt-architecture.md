# AI Prompt 架构模板

> 本文件在 A6 节点调用。
> 小K根据主人描述的 AI 功能，填充这个模板，生成 prompt 架构草稿。

---

## 核心方法论：用工程方法做 Prompt，不靠反复强调

写 prompt 不是靠堆「重要、准确、别写水」这类强调词——那些没用。
有效的 prompt 是工程产物：结构清晰、有靶心、有能力配套、有测试验证。

### 四层结构

**第一层：让 AI 读得懂**
- 结构清晰优先于指令长度，越长不代表越好
- 用标签隔离不同性质的信息：角色、规则、语气、输出格式分开写，不要混在一段话里
- 信息分层之后，AI 才能准确定位每一条指令的作用域

**第二层：给 AI 靶心**
- 明确角色：AI 在这个任务里扮演什么，评判者、提取者、还是生成者？
- 给示例：输出格式给 example，不要只描述——「输出如下格式」远不如直接给一个填好的例子
- 给思考步骤：复杂判断拆成 Step 1→2→3，不要让 AI 一步出结论

**第三层：给 AI 能力**
- 指令解决不了能力问题——「务必计算准确」没用，遇到计算、查数据类任务要给工具
- 能力边界要明确：AI 不该猜的地方，用 Step Zero 门槛检查拦住
- 能力之外的任务，宁愿返回 rejected，不要让 AI 硬猜

**第四层：证明它有效**
- 每次改 prompt 都要跑测试，不能只凭感觉「好像变好了」
- 每次只改一处，改完立即验证，确认解决了失败项，同时检查有没有引入新问题
- 旧的防御性指令在新模型上可能反而干扰输出——定期清理过时「补丁」
- 换模型或升级模型后，必须重新验证，同一 prompt 在不同模型上反应可能不同

## 评估体系（先建立，再写 prompt）

> 没有测试用例，改好改坏都不知道。评估体系是 prompt 工程的地基，不是最后才加的事。

### 三类用例

| 类型 | 说明 | 作用 |
|------|------|------|
| 控制用例 | 典型的正常输入，期望输出明确 | 验证基础功能是否正常 |
| 边缘用例 | 极端情况：最短/最长输入、混合语言、格式异常 | 发现 prompt 的盲区 |
| 边界用例 | 故意写错、空输入、整体不符合条件 | 验证 Step Zero 和兜底校验 |

### 失败项排查逻辑

拿初始 prompt 跑测试，定位失败项之后，按这个顺序排查：

1. **规则冲突**：两条规则在这个输入上产生矛盾 → 明确优先级或合并规则
2. **信息缺失**：prompt 里没有这种情况的处理说明 → 补规则
3. **缺少执行能力**：AI 没有工具或权限完成这个任务 → 加工具，不要加指令

## Prompt 模板结构

```xml
<role>
[AI 在这个任务里的角色，一句话]
</role>

<step_zero>
门槛检查（整体不符合时在入口拦住，不做后续分析）：
- 触发条件：[什么情况整体不符合]
- 触发后返回：{"status": "rejected", "reason": "[说明]"}
</step_zero>

<rules>
判断规则（每条必须明确，不允许模糊措辞）：
- 规则1：[可引用原文的明确条件] → 输出 [字段] = [值]
- 规则2：[明确条件] → 输出 [字段] = [值]
- 边界情况：[edge case] → 输出 [值]
</rules>

<output_format>
输出格式（先定格式，再写规则；这是和前端之间的合同）：

字段定义：
- [field1]: string — 枚举值：[val_a / val_b / val_c]
- [field2]: number — 范围：[0-100]
- [field3]: boolean — true / false

Example（必须给，不能只描述）：
{
  "field1": "val_a",
  "field2": 85,
  "field3": true
}
</output_format>

<thinking_steps>
复杂判断时的思考步骤（让 AI 分步推导，不要一步出结论）：
Step 1：[先做什么]
Step 2：[再做什么]
Step 3：[最后输出]
</thinking_steps>
```

## 前端 Parser 绑定说明

> prompt 改了，这里必须同步更新。parser 改了，也要回来对照这里。
> 任何一边改了没通知另一边，必然出问题。

| prompt 输出字段 | parser 解析逻辑 | UI 行为 |
|----------------|----------------|---------|
| [field1] = val_a | parser → state.xxx = 'A' | [显示/隐藏/跳转] |
| [field2] >= 85 | parser → 高分模式 | [显示夸奖文案] |
| [field2] < 60 | parser → 低分模式 | [显示改进建议] |

**Fallback 策略**（主解析失败时降级，不报错给用户）：
1. 先尝试精确匹配
2. 失败 → 降级到位置解析
3. 再失败 → 返回默认值

---

## 兜底校验

> 前端加这层，不能 100% 依赖 prompt 保证正确性。
> AI 输出可能内部矛盾，前端要能检测并处理。

矛盾检测：
- [field_A] 描述「有」但 [field_B] 给了极低值 → 以 [field_B] 为准 / 触发 fallback
- [其他矛盾情况]

异常处理：
- AI 返回空 → [处理]
- 字段缺失 → [处理]
- 网络超时 → [处理]

## 改 Prompt 的操作规范

- [ ] 每次只改一处
- [ ] 改完立即跑全部测试用例
- [ ] 确认失败项解决了，同时检查有没有引入新问题
- [ ] 改了输出格式 → 必须同步通知前端 parser
- [ ] 清理过时的防御性指令（「务必」「一定要」这类无效强调）
- [ ] 换模型后重新验证所有用例，不假设结果一样
